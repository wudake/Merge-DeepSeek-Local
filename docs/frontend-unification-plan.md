# 三合一项目前端统一化方案

## 一、当前前端技术栈现状

### 1. 主平台前端（运营管理 + 视频脚本提取 + 系统设置）

位于 `frontend/`，是一个统一的 React SPA，运营管理和脚本提取已共享同一套代码。

| 技术 | 版本 | 说明 |
|---|---|---|
| React | 19.2 | + TypeScript 5.7 |
| 构建工具 | Vite 6 | 开发/构建/预览 |
| UI 组件库 | Ant Design 5.24 | 企业级组件库 |
| 路由 | React Router v7 | 兼容 v6 API |
| 状态管理 | Zustand 5 | 全局状态 |
| HTTP 客户端 | Axios 1.8 | API 请求 |
| 日期处理 | Day.js 1.11 | 周视图等日期逻辑 |
| 实时通信 | Socket.io-client 4.8 | 脚本提取任务状态推送 |
| 图表 | @ant-design/charts 2.2 | 数据看板 |
| 样式辅助 | Tailwind CSS 3.4 | 少量使用 |

### 2. 视频自动化前端

位于 `services/video-auto/templates/index.html`，是一个独立页面，通过 iframe 嵌入主平台。

| 技术 | 说明 |
|---|---|
| 纯原生 | HTML + CSS + JavaScript，无框架 |
| 模板引擎 | Flask Jinja2 服务端渲染 |
| 嵌入方式 | iframe 嵌入主平台的 `VideoAutoPage` |

**关键差异**：视频自动化是三个模块中唯一未接入 React 技术栈的部分，也是"前端统一"的核心改造对象。

## 二、前端统一工作量评估

"前端统一"的核心工作量 = **把视频自动化模块从原生 JS 迁移到 React + Ant Design**。

### 功能拆解与工时

| 模块 | 功能点 | 预估人天 |
|---|---|---|
| **页面骨架** | 步骤向导（1下载 → 2配置 → 3完成）、路由切换、整体布局 | 1 |
| **步骤1：获取视频** | 链接下载、本地上传、状态提示、上传成功信息展示 | 1 |
| **步骤2：剪辑配置** | Logo上传/选择、BGM上传/选择/试听、音频音量滑块、TTS配音面板（文案/音色/语速/试听/生成）、视频效果复选框、裁剪时间输入、播放速度/Logo大小/位置滑块 | 3-4 |
| **字幕系统** | 启用手动字幕、TTS同步字幕（互斥逻辑）、字幕内容/时间/样式/字体大小/位置/对齐/描边、**实时效果预览**（9种样式的CSS动态渲染） | 2 |
| **步骤3：结果展示** | 视频预览、下载链接、二维码展示、手机扫码提示 | 1 |
| **日志与状态** | 剪辑进度状态、错误详情、可折叠的终端风格日志渲染 | 1 |
| **API 对接** | 下载/上传/剪辑/TTS/BGM试听等约10+个接口，axios封装 | 1 |
| **联调测试** | 端到端流程验证（下载→剪辑→预览→下载），边界case处理 | 2-3 |

**总计：约 12-16 人天**（一个熟练前端工程师约 2.5-3 周）

## 三、主要风险

### 高风险

#### 1. 功能对等风险
原生 JS 页面里有大量内联 DOM 操作和样式计算，迁移到 React 时容易遗漏：
- **字幕预览**：9种样式（经典黄字、霓虹青光、TikTok风格等）依赖复杂的 `text-shadow` / `background` 动态计算，共60+行原生DOM style赋值，迁移时需逐行对照
- **TTS与手动字幕互斥**：`opt-tts-subtitles` 和 `opt-subtitles` 的互斥逻辑、面板显示隐藏、状态同步，React中需用state重构，容易引入状态不一致bug

#### 2. iframe隔离优势的丢失
当前iframe的好处是video-auto前端和后端在同一个"沙盒"内，Cookie/Session自然共享。合并到React SPA后：
- 需要确保所有API调用都走JWT Bearer token（目前后端已支持，但需前端统一传递方式）
- 文件上传（Logo/BGM/视频）的 `FormData` 处理需要从iframe模式调整到axios模式，Content-Type边界处理可能出问题

### 中风险

#### 3. 步骤向导状态管理
步骤1产生的 `currentNoteId` 必须在步骤2、步骤3中可用，且页面刷新后不应丢失：
- 当前原生页面用全局变量 `let currentNoteId` 存储
- React中如果用纯local state，刷新会丢；用 `localStorage` 或URL query持久化会增加复杂度

#### 4. 向后兼容与用户习惯
运营人员可能已经习惯了当前的操作流程（尤其是字幕样式预览的深色背景、TTS面板的展开动效）。Ant Design的组件风格偏"企业管理后台"，和当前video-auto偏"创意工具"的暗色/卡片化风格差异较大，统一后可能显得"太正经"。

#### 5. 开发期间功能冻结
改造期间（2-3周），视频自动化模块不能同时两个人改（一个改原生、一个改React），否则代码会冲突。这意味着这期间如果视频自动化出现bug，修复和重构会交织，增加回归测试成本。

## 四、降低风险的方案

| 方案 | 说明 | 适合情况 |
|---|---|---|
| **A. 一步到位** | 直接重写整个video-auto前端，2-3周集中开发，一次性替换iframe | 业务淡季、团队人力充足、能接受功能冻结 |
| **B. 渐进迁移（推荐）** | 按步骤分批改造，每完成一个步骤就上线一个，新旧页面并存期间用路由切换 | 业务不能停、需要持续验证、风险厌恶 |
| **C. iframe 包装** | 不改video-auto内部，只在React里做一个"统一外壳"页面，iframe嵌入但加上Boswindor统一Header和底部操作栏 | 追求最低成本、可接受iframe缺陷、时间紧迫 |

### 推荐方案 B 的里程碑

| 阶段 | 交付内容 | 周期 |
|---|---|---|
| **Phase 1** | 改造"剪辑完成"结果页（视频预览 + 下载 + 二维码） | 1周 |
| **Phase 2** | 改造"获取视频"页（链接下载 + 本地上传） | 1周 |
| **Phase 3** | 改造最复杂的"剪辑配置"页（所有配置项 + 字幕预览） | 1-2周 |
| **Phase 4** | 移除iframe，完全接入React路由和统一Header | 2天 |

渐进迁移的优势：每1-2周就有可交付的价值，随时可以暂停或调整优先级。

---

## 五、决策建议

| 问题 | 建议 |
|---|---|
| 如果业务不能停 | 选渐进迁移（方案B），分4个Phase交付 |
| 如果只有1个人力 | 选渐进迁移，Phase 1和2可以合并，总周期约3周 |
| 如果时间极紧（1周内） | 先选方案C（iframe包装）做临时方案，后续再迁 |
| 如果追求长期最优 | 一步到位（方案A），一次性解决iframe技术债务 |
